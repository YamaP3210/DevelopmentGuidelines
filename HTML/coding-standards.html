概要

このページは、C# のコーディング規約を示します。

命名、表記、初期化、コレクション、ツール、非同期、文字列、例外、論理演算子、delegate、イベントを扱います。

命名

命名は、具体的な書式ルールと、判断の考え方を分けて扱います。

規則

名前空間

ルート名前空間はケース識別子から始めます。ケース内フォルダ構造と完全一致させる必要はなく、基本はフォルダに沿って考えつつ、カテゴライズ、利便性、可読性を優先して組みます。

たとえば、ケース識別子が XXX なら、XXX.Client.UI、XXX.Client.Tools、XXX.Server.Core のように、役割や配置で分ける形が考えられます。

よほどの理由がない限り、ケース間で名前空間を共有しません。XXX を共有するということは XXX 内で完結させる考え方であり、別のケースから関与するなら YYY のように別の名前空間で扱います。

クラス名

クラス名はケース識別子の直後に PascalCase を続けます。_ のような連結記号は使いません。たとえば、ケース識別子が XXX で、略語を含む接続名なら XXXTCPConnection のように続けます。

型名

class、struct、enum の型名は、クラス名と同じくケース識別子の直後に PascalCase を続けます。

インターフェース

インターフェース名は I + ケース識別子 + PascalCase で続けます。属性型は Attribute で終えます。

プロパティ

プロパティは PascalCase で書きます。プロパティは状態・設定・パラメータを表すために用い、処理をさせるためのものとして扱いません。private プロパティは、それがプロパティである理由が明確な場合に使い、必要性が薄いならフィールドの使用を検討します。詳細はフィールド・プロパティの使い分けで扱います。

フィールド

フィールドは _ + camelCase で書きます。公開フィールドは原則として使わず、必要がある場合でも慎重に扱います。

メソッド・イベント・定数

メソッドは動詞で、イベントと定数は PascalCase を使います。ローカル変数、引数、private フィールドは camelCase を使います。

識別子の基本

識別子は先頭に文字または _ を使い、必要なら @if のように verbatim identifier を使えます。

備考

ケース識別子

ケースには識別子があり、そこから生み出されたものの接頭辞として使うことを原則にします。ファイル名、クラス名、名前空間、設定名などの先頭に付けますが、ファイル名固定のライブラリやリネームできない資産のような技術的制約がある場合は例外を認めます。

略語

略語・頭文字語は大文字のまま扱い、他の語と連結するときも Xxx のように通常の PascalCase に丸めません。たとえば、ID は Id にせず ID のまま、TCP は TCP のまま、UI は UI のまま扱い、名前空間なら XXX.ID、XXX.TCP、XXX.UI のように続けます。略語の意味を保ったまま続けます。ただし、言語規約や外部仕様、技術的制約に抵触する場合は例外を認めます。

命名の優先度

まず意味が伝わることを優先し、そのうえで既存の .NET 慣例と整合させます。プロジェクト固有の命名が必要な場合は、例外として扱います。

表記と初期化

型表記

型は string や int などの C# キーワードを優先し、var は型が明確な場合に使います。

初期化

文字列補間、raw string literal、new()、オブジェクト初期化子、required を適切に使います。

コレクション

配列やコレクションの初期化は collection expressions を優先します。複数の要素を扱う場面では、読みやすさを崩さない範囲で LINQ を使います。

ツール

.editorconfig と code analysis を使って規約を整えます。Visual Studio の編集時設定だけに依存しません。

非同期

I/O を伴う処理は async / await を使い、同期ブロックを避けます。キャンセル可能な処理には CancellationToken を渡します。

文字列

短い連結は文字列補間を使い、ループ内の大量連結は StringBuilder を使います。raw string literal も必要に応じて使います。

例外処理

捕捉する例外は扱えるものに絞り、再送出では情報を失わない形を優先します。using で済むなら、解放処理のためだけに try-finally を使いません。

論理演算子

比較では && と || を使い、ビット演算との混同を避けます。短絡評価が必要な場面では、評価順も意識します。

delegate と func/action

単純な処理は Func<> / Action<> を使い、不要な delegate 型の追加は避けます。独自 delegate が必要な場合は、公開契約として意味を持たせます。

イベント

解除不要なら lambda で書き、後で外す必要があるなら明示的なメソッドを使います。イベントの購読と解除の関係が追える形を優先します。
