概要

独自MVVM構成、レイヤード構成、フォルダ構成など、設計の軸となるアーキテクチャを解説します。

独自MVVM

当開発ガイドラインでは、設計パターンとして MVVM を採用しています。ただし、標準的な MVVM をベースにカスタマイズした独自MVVMパターンを採用します。このセクションの内容は、標準的な MVVM パターンの構成と役割を理解している前提で記述します。

この独自MVVMは、Unity における GameObject と Component の構成を前提として整理します。

構成対応

Model

Model は、一般的な MVVM の Model に相当する層です。

データと業務処理を担当し、UI アクションを契機として Coordinator から処理要求を受け、必要な処理を実行してデータを更新します。

更新結果は Coordinator を通じて Widget に通知します。また、複数の Coordinator から利用され、必要に応じて他の Model とも連携します。

Coordinator

Coordinator は、一般的な MVVM の ViewModel に相当する層です。

1 つ以上の Widget を担当し、Model から通知されたデータやイベントを受け取り、担当する Widget に必要な情報を通知します。また、UI アクションに応じて Model に処理を依頼します。

必要に応じて他の Coordinator や複数の Model とも連携します。Widget 間の連携は、必ず Coordinator を介して行います。

注記: Coordinator は、Widget を直接の表示単位として束ね、画面全体の振る舞いを調整する中間層です。Widget は末端の UI 機能単位として個別の入出力や表示制御を担当し、Coordinator はそれらの担当割り当てや相互連携をまとめます。

Widget

Widget は、複数の Component をまとめて 1 つの機能単位として扱う層です。

パネル、入力エリア、一覧表示領域など、ひとまとまりの UI 機能を表し、Coordinator から通知されたイベントやデータに応じて、担当する Component 群へ処理を振り分けます。

Widget 同士は直接連携せず、必要な連携は担当する Coordinator を介して行います。

Component

Component は、View を構成する末端単位です。

単体の GameObject、または単体の機能を提供する複数の GameObject をまとめた構成単位として扱い、ボタン、入力欄、ラベル表示、一覧の 1 要素など、個別の UI 機能を担当します。

Component 同士は直接連携せず、必要な連携は必ず担当する Widget を介して行います。

プロジェクト・フォルダ構成

Unity プロジェクトにおけるフォルダ構成は、Assets フォルダ配下のみを記述対象とします。

基本構成

プロジェクト固有の Assets は、プロジェクト名のフォルダを作成し、その中にすべて格納します。たとえば XXXProject という名前のプロジェクトであれば、プロジェクト固有のリソースやコードは Assets/XXXProject/ 配下にまとめます。

Assets/XXXProject/ 配下には、基本的に次のフォルダを配置します。 

Assets/
└─ XXXProject/
   ├─ Resources/
   ├─ Scenes/
   ├─ Scripts/
   ├─ Prefabs/
   └─ その他プロジェクト固有のフォルダ

Scripts フォルダ

Scripts フォルダには、アプリケーションのコード資産を機能別に分類して配置します。

注記: Core はアプリケーション共通の基幹機能を置くフォルダです。Model はその基盤を利用して、担当領域のデータと状態を管理するフォルダです。両者は役割が近く見えますが、Core は共通基盤、Model は担当領域の実体として分けます。 
Assets/
└─ XXXProject/
   └─ Scripts/
      ├─ Core/
      ├─ Models/
      ├─ Coordinators/
      ├─ View/
      ├─ Tools/
      ├─ Classes/
      ├─ Extensions/
      ├─ Data/
      └─ Database/

Core

Core は、アプリケーションの基幹部として動作するクラス群を配置するフォルダです。

内容に固定の決まりはなく、アプリケーションに応じた基幹部として扱います。たとえば Model や Coordinator の基底クラス、通信やログを扱うクラスなど、MVVM の枠外でアプリケーションの土台として働くクラスを配置します。

Core は、他の構成要素の補助ではなく、それ自体で基幹機能を担う単位として構成します。

構築例

Core の内部構成は、アプリケーションの種類によって変わります。そのため、ここでは代表例として Unity プロジェクトにおける構築例を示します。

Unity

Unity プロジェクトでは、Core にアプリケーション共通の基幹機能を集約します。通信、ログ、設定、共通ユーティリティなど、各 Model や Coordinator が利用する機能をこの層に置きます。

Core は共通基盤として独立して機能し、Model や Coordinator から利用されます。具体的な内部分割は Unity アプリケーションごとに調整しますが、共通処理を個別の Model に持たせず、基盤側へ寄せる方針を基本とします。

Models

Models は、アプリケーション内で Model として扱うクラス群を配置するフォルダです。

Model は、担当領域のデータと状態を管理する層です。Core が提供する共通機能を利用しながら、その領域固有の振る舞いを持ちます。

Coordinators

Coordinators は、アプリケーション内で Coordinator として扱うクラス群を配置するフォルダです。

View

View は、Widget や Component を配置するフォルダです。

構成によっては、XAML、HTML、CSS、JavaScript など、表示や画面構成に関わるファイル群をここに配置する場合もあります。

Tools

Tools は、MVVM に該当しないクラス群を配置するフォルダです。

アプリケーションによっては、Tools と Utilities などに分けて、役割ごとにフォルダ構造を細分化する場合もあります。

Classes

Classes は、カテゴリ分けが難しい雑多な機能を持つクラス群を配置するフォルダです。

Extensions

Extensions は、拡張メソッドを定義するクラス群を配置するフォルダです。

Data

Data は、データモデルとして扱うクラス群を配置するフォルダです。

Database

Database は、DB を扱うアプリケーションで、関連するクラス群を配置するフォルダです。

Scene と Hierarchy

2D アプリケーション開発では、Scene と Hierarchy の初期状態も規約として定めます。

初期 Scene

Unity の 2D プロジェクト初期 Scene には、基本的に Main Camera と 2D 用のライティングオブジェクトが配置されます。テンプレートによっては Grid などの補助オブジェクトが追加される場合があります。

Hierarchy 方針

Hierarchy は、画面の構成要素と役割が分かるように整理します。初期状態の Scene に必要なオブジェクトだけを置き、以後の画面構成は Widget 単位で追加します。

画面要素の実体は GameObject と Component によって構成し、Coordinator はそれらの状態と連携を管理します。
