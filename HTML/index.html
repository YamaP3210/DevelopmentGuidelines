この文書は、Microsoft Learn の C# コーディング規約をベースに、プロジェクト共通の標準規約として読めるように整理したものです。 ここではまだ独自ルールを足さず、標準側の方針をそのまま土台として残します。

適用範囲

新規実装、改修、レビュー、保守、サンプルコード、共通部品の記述に適用します。

個別案件で追加の制約が必要な場合は、この文書を基準に差分として追記します。

方針

  • 可読性、整合性、保守性を優先する。
  • 既存コードと整合する書き方を基本にする。
  • コードは学習資料としても使える水準を目指す。
  • 複雑さを増やす記法は避ける。
  • 意図が分かる名前と構造を重視する。
  • 最新の C# 機能は、明快さを損なわない範囲で採用する。

ツールと分析

Microsoft は、Common C# code conventions の中で、code analysis と .editorconfig を用いて規約を強制することを案内しています。

この文書でも、最終的には IDE と CI で自動検査できる規約を前提にします。

命名規約

Microsoft Learn の C# identifier naming rules and conventions に沿って、基本は次の方針を採ります。

要素 規約
型、名前空間、公開メンバーPascalCase
private / internal フィールドcamelCase かつ必要に応じて接頭辞を付ける
ローカル変数、メソッド引数camelCase
インターフェースI で始める
属性型Attribute で終える
定数PascalCase
型パラメータ意味が伝わる名前を使う。単独なら T も可

略語は、広く通用するもの以外は避けます。

言語ガイドライン

  • 利用可能なら、現代的な C# 機能を使う。
  • 古い構文は、互換性や明確な理由がある場合のみ使う。
  • 汎用的な例外捕捉は避け、扱える例外だけを捕捉する。
  • I/O バウンド処理には async / await を使う。
  • 文字列、数値、型名は、可読性が高い標準表現を使う。
  • var は型が明確に推測できる場合に限る。

文字列

  • 短い文字列連結には文字列補間を使う。
  • ループ内で大量の文字列を連結する場合は StringBuilder を使う。
  • 必要なら raw string literals を使う。
  • 位置指定の補間より、式ベースの補間を優先する。
string displayName = $"{lastName}, {firstName}";
var builder = new StringBuilder();
builder.Append(phrase);

初期化

  • record の primary constructor は、公開プロパティとして扱われる前提で命名する。
  • class / struct の primary constructor parameter は camelCase を使う。
  • 必須初期化が必要なら required を検討する。
  • オブジェクト初期化子を使って、代入の散在を避ける。

配列とコレクション

  • コレクション初期化には collection expressions を使う。
  • 戻り値としては、用途に応じて IEnumerable<T>、IReadOnlyList<T>、List<T> を使い分ける。
  • LINQ は読みやすさを損なわない範囲で使う。

デリゲート

  • 可能なら Func<> と Action<> を使う。
  • 独自 delegate 型を作るのは、意味を持つ公開契約が必要な場合に限る。
  • デリゲート呼び出しは、シグネチャが分かる形で書く。

例外処理

  • 処理できない例外は捕捉しない。
  • 再送出は throw; を使う。
  • try-catch より、using や設計で解決できるならそちらを優先する。
  • 例外メッセージには、調査に必要な文脈を含める。

演算子

  • 比較では && と || を使う。
  • new は省略可能な場合、簡潔な記法を使う。
  • オブジェクト初期化子で見通しをよくする。

イベント

解除不要なイベントハンドラには lambda を使い、後で外す必要があるものは明示的なメソッドを使います。

static メンバー

static メンバーはクラス名で参照し、基底型の static を派生型名で飾る書き方は避けます。

LINQ

  • クエリ変数名は意味が分かる名前にする。
  • 匿名型のプロパティ名は PascalCase を保つ。
  • 複雑なクエリは分解し、読みやすさを優先する。
  • 絞り込みは後続の句より先に行う。

レビュー観点

  • 命名が意味を表しているか。
  • 責務が分かれすぎたり、逆に肥大化したりしていないか。
  • 例外、null、安全性、非同期の扱いに無理がないか。
  • 変更の影響範囲に対してテストが足りているか。
  • 標準規約からの逸脱がある場合、その理由が明示されているか。
