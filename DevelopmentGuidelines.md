# DevelopmentGuidelines

## 始めに

- この文書は、`DevelopmentGuidelines` プロジェクトの成果物である開発ガイドラインを構築、編集、評価する際の方針を定める。
- この文書は、成果物の目的、利用者像、構成、記述、記載形式を判断するための基準を定める。

## 目的

- このプロジェクトの最終成果物は、システム・ソフトウェア開発チーム内の利用者が参照・利用するための、HTML形式の開発ガイドラインWebページである。

## 利用者像

- HTML の開発ガイドラインの主読者は、利用者であるシステム・ソフトウェア開発者である。
- 成果物は、利用者の視点で、ガイドラインとして読み、利用できる内容と表現で記述しなくてはならない。

## 成果物方針

- 開発ガイドラインは、人が開発規約を確認するための公開資料として扱うことを原則として優先して考えれば良い。
- 冒頭は、文書の存在理由を長く説明するより、規約本文へすぐ入れる構成にすることを原則として優先して考えれば良い。
- セクションは、C#、Unity、Visual Studio、基本アーキテクチャのように、実務上の判断単位で分けることを原則として優先して考えれば良い。
- 内容が増えてきたら、1 ページに集めず、実務上の判断単位ごとにページを分けることを原則として優先して考えれば良い。
- 各章は、主読者である利用者が見たときに「次に何を決めればよいか」が分かる構造にしなくてはならない。

## 記述方針

- 具体例は理解の補助として使い、規約の本体は抽象度の高い原則で表すことを原則として優先して考えれば良い。
- 例示に使う固有名詞は、文脈に関係するものだけに絞り、無関係なケース名や識別子を使ってはならない。
- 各セクションの説明は、その章を読むことで何が分かるかがすぐ伝わる程度に留めることを原則として優先して考えれば良い。
- HTML の概要やページ紹介は、読者がそのページで何を得られるかを示す文章にしなくてはならない。
- HTML のスタイル・フォーマット関係の記述は、コードブロックでサンプルを示すことを原則として優先して考えれば良い。

## 記載形式

- 記載形式に関する方針は、利用者が読みやすく、内容を参照しやすい形を優先して定めなくてはならない。
