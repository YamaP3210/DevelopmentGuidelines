# DevelopmentGuidelines

この文書は、`DevelopmentGuidelines` プロジェクトを扱う AI が、作業判断の材料として使う方針・考え方をまとめる。
個別の命令や細かな禁止事項を積み上げるためではなく、文書全体を一貫した軸で扱うことを優先して考えれば良い、という判断の基準を残す。

## 判断の軸

- 人間向けの成果物は、実際に読む人がそのまま使える公開資料として扱うことを優先して考えれば良い。
- AI 向けの補助的な理解は、成果物の本文に混ぜず、必要な場合だけ別文書で保持することを優先して考えれば良い。
- 文章は、単発のルールよりも、読者像と文書目的に沿った上位原則で整理することを優先して考えれば良い。
- 役割の重複や説明の重複は避け、各文書が自然に分担する構成にすることを優先して考えれば良い。

## 命名の規則

- class、struct、enum の型名は、ケース識別子の直後に PascalCase を続け、`_` のような連結記号は使わなくて良い。
- インターフェース名は `I` + ケース識別子 + PascalCase で続ければ良い。
- プロパティは PascalCase で書き、private プロパティは、それがプロパティである理由が明確な場合に使い、必要性が薄いならフィールドを検討すれば良い。
- フィールドは `_` + camelCase で書き、公開フィールドは原則として使わなくて良い。
- 名前空間のルートはケース識別子から始め、ケース内フォルダ構造と完全一致させることは必須にしなくて良い。基本はフォルダに沿って考えつつ、カテゴライズ、利便性、可読性を優先して組めば良い。
- 例としては、`XXX.Client.UI`、`XXX.Client.Tools`、`XXX.Server.Core` のように、ケース識別子の下で用途や役割ごとに名前空間を分けて考えれば良い。
- よほどの理由がない限り、ケース間で名前空間を共有しない。`XXX` を共有するという考え方なら `XXX` 内で完結させ、別のケースから関与するなら `YYY` のように別の名前空間で扱えば良い。
- 略語・頭文字語は大文字のまま扱い、他の語と連結するときも `Xxx` のように通常の PascalCase に丸めなくて良い。
- `ID` は `Id` にせず `ID` のまま、`TCP` は `TCP` のまま、`UI` は `UI` のまま扱い、名前空間なら `XXX.ID`、`XXX.TCP`、`XXX.UI` のように続け、クラス名なら `XXXTCPConnection` のように続ければ良い。
- 識別子は先頭に文字または `_` を使い、必要なら `@if` のように verbatim identifier を使えば良い。

## 命名の備考

- ケースには識別子があり、そこから生み出されたファイル名、クラス名、名前空間などの接頭辞として使うことを原則にし、ファイル名固定のライブラリやリネームできない資産のような技術的制約がある場合は例外を認めれば良い。
- 文字数削減よりも、意味が伝わる命名を優先して考えれば良い。
- 一般的に違和感なく使われる略称は、そのまま使ってよい。
- `DataBase` を `DB` とするような、意味が通る略称は許容してよい。
- ループカウンターなどに `i` を使わず、何を数えているかが分かる名前を優先して考えれば良い。
- プロジェクト固有の命名が必要な場合は、例外として扱えば良い。

## 作業方針

- 開発ガイドラインは、人が開発規約を確認するための公開資料として扱うことを優先して考えれば良い。
- 文書の冒頭は、長い説明よりも本題へ進める構成にすることを優先して考えれば良い。
- セクション構成は、C#、Unity、Visual Studio、基本アーキテクチャのように、実務で判断しやすい単位で考えることを優先して考えれば良い。
- 内容が増えてきたら、1 ページに詰め込まず、判断単位ごとにページを分けることを優先して考えれば良い。
- 章や見出しは、読者が次に何を確認すればよいかが自然に分かる形に整えることを優先して考えれば良い。
- HTML の概要やページ紹介は、読者がそのページで何を得られるかを示す文章にし、制作者側の思考や運用意図は備考・特記事項として分けることを優先して考えれば良い。
- 作業方針は定期的に見返し、局所的な文言の微修正より優先して考えれば良い。
- HTML のスタイル・フォーマット関係の記述は、コードブロックでサンプルを示すことを優先して考えれば良い。
- クラス内部の書式は、カテゴリの切り替えは 5 行空け、同一カテゴリ内の項目は 3 行空け、メソッド内の処理の塊は 1 行空けで考えれば良い。
- テキストファイルの改行は Visual Studio 標準に合わせ、CRLF を基準にすれば良い。
- 個別の注意点は、その場しのぎの規則として増やしすぎず、上位の方針に吸収できるかを先に考えることを優先して考えれば良い。

## 記述の考え方

- 読者が知りたいのは、何を守るか、どう実装するか、どの基準で判断するかであると考える。
- そのため、説明のための説明を増やしすぎず、判断に必要な情報を優先して考えれば良い。
- 具体例は理解の補助として使うが、細則の代替にはしない。
- 方針どうしが衝突しそうなときは、細部ではなく、より上位の意図に立ち返って整理することを優先して考えれば良い。

## 維持方針

- 新しい記述を加えるときは、既存の方針との整合を先に見ることを優先して考えれば良い。
- 役割が重なる文書が出た場合は、最も自然に読める場所へ寄せることを優先して考えれば良い。
- この文書は、細則を増やすためではなく、AI の判断軸を保つために更新することを優先して考えれば良い。
