# C# 開発規約

本書は、システム開発プロジェクトにおける C# 実装の共通規約を定める。
新規実装、改修、レビュー、保守の判断基準は、原則として本書に従う。

## 1. 基本方針

- 可読性、保守性、拡張性を優先する。
- 仕様が不明確な場合は、実装を先行させず確認する。
- 1 つのメソッド、クラス、モジュールは責務を絞る。
- 例外的な処理は局所化し、標準的な処理経路を簡潔に保つ。
- 既存コードの流儀と矛盾する独自実装は避ける。

## 2. 命名規約

- 型名、メソッド名、プロパティ名、公開メンバーは PascalCase を用いる。
- ローカル変数、引数、private フィールドは camelCase を用いる。
- インターフェース名は `I` で始める。
- 非同期メソッド名は `Async` で終える。
- 真偽値は `Is`、`Has`、`Can`、`Should` など意味が明確な接頭辞を使う。
- 略語は一般的なもの以外を避ける。やむを得ず使う場合はプロジェクト内で統一する。

## 3. クラス設計

- クラスは単一責務を基本とする。
- 依存は可能な限りコンストラクタで注入する。
- 公開 API は最小限にする。不要な `public` は増やさない。
- 継承より合成を優先する。
- DTO、Entity、Domain、Service、Repository の責務を混在させない。
- データ保持用の型は必要に応じて `record` も検討する。

## 4. メソッド設計

- 1 メソッド 1 目的を原則とする。
- ネストが深い処理は分割する。
- 引数は必要最小限にする。3 つを超える場合は集約を検討する。
- 戻り値は、呼び出し側の判断に必要な情報を含める。
- `out` 引数は原則避ける。
- 1 メソッド内で I/O、変換、検証、ビジネスロジックを混在させない。

## 5. 非同期処理

- I/O を伴う処理は原則として非同期化する。
- `async` メソッドでは `await` を適切に使い、不要な同期ブロックを避ける。
- `Task.Result` や `Task.Wait()` は原則使用しない。
- `async void` はイベントハンドラ以外で使わない。
- キャンセル可能な処理には `CancellationToken` を渡す。

## 6. 例外処理

- 例外は握りつぶさない。
- 例外は意味のある境界で捕捉し、原因と文脈を保って再送出する。
- 想定される業務エラーと、想定外のシステム例外を区別する。
- `throw ex;` は使わず、再送出は `throw;` を用いる。
- 入力不備は可能な限り早い段階で検証する。

## 7. Null と初期化

- null 許容参照型を有効にすることを前提とする。
- `null` を許容する場合は、意味と扱いを明示する。
- `!` 演算子の多用は避ける。必要な場合でも根拠を明確にする。
- コレクションは未使用なら空コレクションを返す。

## 8. コレクションと LINQ

- 目的が明確な場合のみ LINQ を使う。
- 複雑な LINQ は分割し、読みやすさを優先する。
- 副作用のある処理を LINQ に埋め込まない。
- 配列、`List<T>`、`IEnumerable<T>` の使い分けを意識する。

## 9. 文字列とフォーマット

- 文字列連結より補間文字列を優先する。
- ログや例外メッセージは、原因特定に必要な情報を含める。
- 日付、数値、通貨、区切り文字の扱いは文化依存を明示する。

## 10. ログ

- ログは後追い調査に必要な粒度で残す。
- 重要な分岐、外部 I/O、例外発生点は記録対象とする。
- 個人情報、秘密情報、認証情報は出力しない。
- ログレベルは次の基準を守る。
  - `Trace` / `Debug`: 開発・調査用
  - `Information`: 通常の業務イベント
  - `Warning`: 想定内だが注意が必要
  - `Error`: 処理失敗
  - `Critical`: システム継続に重大な影響

## 11. 入出力と外部連携

- 外部システムとの境界は専用の層に分離する。
- 変換処理は境界で行い、内部ロジックに外部仕様を持ち込まない。
- リトライ、タイムアウト、冪等性は必要に応じて設計に含める。
- API、DB、ファイル、メッセージングはそれぞれの失敗パターンを考慮する。

## 12. テスト

- 新規実装や修正には、影響範囲に応じたテストを追加する。
- 正常系だけでなく、境界値、異常系、例外系も確認する。
- ロジックの中心部分はユニットテストで担保する。
- バグ修正には、再発防止のテストを優先する。
- テスト名は、前提・操作・期待結果が分かるようにする。

## 13. コメントとドキュメント

- コードが読めば分かる内容はコメントしない。
- なぜそうしているか、制約は何か、将来の注意点を残す。
- 公開 API や重要な処理には利用条件を明記する。
- 仕様変更時は関連ドキュメントも更新する。

## 14. フォーマット

- フォーマットはプロジェクトで統一する。
- インデント、改行、波括弧の位置、空行のルールを揃える。
- 自動整形ツールがある場合はそれを正とする。
- 手作業で整形ルールを壊さない。

## 15. レビュー基準

- 命名が意味を表しているか確認する。
- 責務が分散しすぎていないか確認する。
- 例外、ログ、非同期、null、安全性に問題がないか確認する。
- 変更の影響範囲に対してテストが十分か確認する。
- 既存実装との差分が必要最小限か確認する。

## 16. 禁止事項

- `Task.Result`、`Task.Wait()` の安易な使用
- `async void` の乱用
- 例外の握りつぶし
- `throw ex;`
- 不要な `public` の追加
- ログへの秘密情報出力
- 責務が肥大化したクラスやメソッドの放置

## 17. 運用原則

- 規約は固定ではなく、プロジェクト運用に合わせて改訂する。
- 例外運用を作る場合は、理由と適用範囲を明示する。
- 迷った場合は、簡潔で検証しやすい設計を優先する。

---

本書は共通規約の基準文書とし、個別案件で必要な追加ルールがある場合は別紙で定義する。
